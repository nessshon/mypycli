# i18n для mypycli

## Цель

Ввести механизм локализации в библиотеку `mypycli` так, чтобы:

- Все пользовательские строки встроенной консоли, CLI-команд и интерактивных промптов можно было перевести.
- Приложения, построенные на `mypycli`, использовали тот же механизм для своих строк.
- Библиотека поставляла собственные переводы и была расширяема силами пользователя.
- Поведение было строгим: промах ключа — ошибка, а не тихая деградация.

## Что переводится

- Вывод в консоль (`print`, `console.print_*`) у кода, обращённого к пользователю.
- Welcome/goodbye-баннеры консоли.
- `Command.description` — текст, показываемый в `help`.
- Тексты интерактивных промптов (`Input`, `Secret`, `Confirm`, `Select`, `Multiselect`).
- Префиксы в сообщениях об ошибках, которые достигают пользователя (например `"Install failed for {name}: {exc}"` — переводим `"Install failed for {name}"`, `{exc}` остаётся оригинальным).

## Что не переводится

- Записи логов (`logger.info/debug/warning/error`) — диагностика на едином языке.
- Оригинальный текст исключений (`str(exc)`).
- `Command.usage` — короткая форма аргументов команды.

## Формат каталогов

Один файл на язык: `locales/<lang>.yml`. Формат — YAML, парсер — `pyyaml` + `yaml.safe_load`. Без комментариев: при `locales sync` файлы пересохраняются целиком.

Структура внутри файла — вложенные словари. Верхний ключ определяет владельца ключа:

```yaml
# Секция библиотеки — управляется mypycli:
mypycli:
  console:
    welcome: "Welcome to {label}!"
    goodbye: "Exiting {label}. Bye!"
  install:
    confirm: "Continue installation?"
    failed: "Install failed for {name}"

# Секции пользователя — произвольные корневые ключи, произвольная вложенность:
dashboard:
  title: "Dashboard"
modules:
  ton_storage:
    status: "TON Storage: {state}"
```

### Внутреннее представление

При загрузке каталог сплющивается в плоский словарь с точечными ключами:

```python
{
    "mypycli.console.welcome": "Welcome to {label}!",
    "mypycli.install.failed": "Install failed for {name}",
    "dashboard.title": "Dashboard",
    "modules.ton_storage.status": "TON Storage: {state}",
}
```

Коллизия между листом и веткой (`foo: "bar"` и `foo: { baz: "..." }` в одном файле) — `ValueError` при загрузке.

### Интерполяция

В шаблонах используется `str.format`-синтаксис: `{name}`, `{label}`, и т.д. Вызов `_(key, **kwargs)` форматирует шаблон, подставляя переданные `kwargs`.

- Нет плейсхолдеров в шаблоне — `kwargs` игнорируются.
- Плейсхолдер есть, а `kwargs` не передан — `KeyError` от `str.format` (ошибка вызова, не маскируется).

Плюрализация встроенно не поддерживается — пользователь реализует через отдельные ключи или композицию при необходимости.

## Загрузка и резолв языка

Загрузка ленивая: `Translator` при инициализации сканирует `locales/*.yml` для множества `available_languages`, содержимое не читает. Фактическое чтение yaml происходит в `set_language()`, и только для выбранного языка. В памяти одновременно находится один каталог.

Смена языка в рантайме не поддерживается. Изменения языка требуют перезапуска приложения.

### Алгоритм резолва в `Application.start()`

```
1. available = translator.available_languages()
   Если пусто — RuntimeError (нет ни одного *.yml в locales/).

2. Если db.language задан и входит в available:
      translator.set_language(db.language)
      return

3. detected = _parse_lang_env(os.environ.get("LANG", ""))
   Если detected непуст и входит в available:
      db.language = detected   # auto-save
      translator.set_language(detected)
      return

4. Fallback: "en" если есть среди available, иначе первый доступный.
      db.language = chosen
      translator.set_language(chosen)
```

`_parse_lang_env` отрезает часть до первого `_` или `.`, затем приводит к нижнему регистру и проверяет что результат — две латинские буквы (`^[a-z]{2}$`). Иначе возвращает `""`. Примеры: `en_US.UTF-8` → `en`; `ru_RU` → `ru`; `C`, `POSIX`, `C.UTF-8`, `""` → `""`.

### Поле `language` в БД

`language` — framework-owned extra на `Database`, хранится как `extras["language"]` (тот же механизм что у `debug`). Реализуется как property с getter/setter, стартовое значение `""`.

```python
# mypycli/database/store.py — Database class
@property
def language(self) -> str:
    """Framework-owned language code persisted under the ``language`` extras key."""
    return str(self.get_extra("language") or "")

@language.setter
def language(self, value: str) -> None:
    self.set_extra("language", str(value))
```

При `load(auto_create=True)` инициализируется в `""` аналогично `debug`.

После первого `app.start()` всегда содержит валидный код, существующий среди `locales/`. Пользователь может сменить язык через `db set language ru` — сохраняется, эффект на следующий старт.

## API

### `Translator`

Класс живёт в `mypycli/i18n/translator.py`:

```python
class Translator:
    def __init__(self, locales_dir: Path | str) -> None:
        """Сканирует *.yml в locales_dir. Содержимое не читает."""

    def available_languages(self) -> set[str]:
        """Коды языков по именам файлов (без расширения)."""

    def set_language(self, lang: str) -> None:
        """Читает locales/<lang>.yml в память.
        Если lang не в available_languages — ValueError."""

    def __call__(self, key: str, **kwargs) -> str:
        """Lookup полного точечного ключа.
        LookupError если не найден.
        KeyError от format если передан недостающий плейсхолдер.
        RuntimeError если set_language ещё не был вызван."""

    @property
    def language(self) -> str | None:
        """Текущий язык или None если не инициализирован."""
```

### Использование пользователем

Инстанс создаётся один раз, передаётся в `Application`:

```python
from mypycli import Application
from mypycli.i18n import Translator

_ = Translator("./locales")

app = Application(
    db_schema=MySchema,
    work_dir="./data",
    translator=_,
)
app.run()
```

Имя переменной произвольное (`_`, `translate`, `t`). `app.translator` возвращает ту же ссылку.

### Использование внутри библиотеки

Отдельный приватный модуль `mypycli/i18n/internal.py`:

```python
_active: Translator | None = None

def bind(t: Translator) -> None:
    global _active
    _active = t

def _(key: str, **kwargs) -> str:
    if _active is None:
        raise RuntimeError("Translator not bound — app.start() не выполнялся")
    return _active(f"mypycli.{key}", **kwargs)
```

Весь код библиотеки пишет:

```python
from mypycli.i18n.internal import _
_("install.failed", name=module.name)
```

Ключ автоматически префиксуется `mypycli.`, так что реальный lookup — `mypycli.install.failed`.

`Application.start()` вызывает `i18n.internal.bind(self.translator)` после резолва языка. `mypycli.i18n.__init__.py` не реэкспортирует `internal` — пользователю он не виден.

## CLI-команды библиотеки

Библиотека публикует собственную CLI-точку входа `mypycli` (отдельно от CLI приложений). Три команды под группой `locales`:

### `mypycli locales init`

Запускается в корне проекта. Читает `mypycli/locales/*.yml` (поставляются библиотекой через `importlib.resources`), для каждого языка создаёт в текущей директории `./locales/<lang>.yml`. Содержимое нового файла — только секция `mypycli:` с дефолтами.

Существующие файлы не перезаписываются (skip + сообщение). Флагов нет. Пользователь удаляет файлы ненужных ему языков; остаток определяет `supported_languages` приложения.

### `mypycli locales sync`

Для каждого `./locales/<lang>.yml`:

1. Читает yaml.
2. Заменяет подветку `mypycli:` на свежую из `mypycli/locales/<lang>.yml`.
3. Прочие top-level ключи (пользовательские) не трогает.
4. Пересохраняет файл.

Если пользовательский язык отсутствует в библиотеке (например `fr.yml` создан вручную) — warning, файл пропускается.

### `mypycli locales check`

Проверяет согласованность каталогов в `./locales/`:

1. Для каждого `./locales/<lang>.yml`, если в библиотеке есть `mypycli/locales/<lang>.yml`: секция `mypycli:` должна точно совпадать (по множеству ключей) с библиотечной. Иначе — ошибка «out of sync, run `mypycli locales sync`». Если в библиотеке нет соответствия (например пользователь создал `fr.yml` вручную) — проверка `mypycli:` секции для этого файла пропускается с информационной строкой.
2. Между всеми пользовательскими top-level ключами: множество ключей в разных языках должно быть идентичным. Иначе — ошибка «missing keys in `<lang>`: ...».

Exit 0 — всё чисто; exit 1 — есть проблемы. Команда предназначена для CI.

## REPL-поведение

Команда `language` и рантайм-переключение языка не вводятся. Смена языка — через обычный `db set language <code>`:

- Пре-валидация: `<code>` должен быть в `app.translator.available_languages()`. Иначе красное сообщение об ошибке со списком доступных, без записи в БД.
- Запись выполняется стандартным `cmd_db_set`.
- После успешной записи `language` или `debug` — жёлтое сообщение «Restart the application for the change to take effect.» (тот же стиль что и другие подсказки консоли, через `colorize_text(..., Color.YELLOW)`).

`cmd_db_set` awareness ограничен этими двумя полями; остальные поля обрабатываются без специальной логики.

## Обработка ошибок

| Ситуация | Поведение |
|---|---|
| `_("key")` — ключа нет в текущем языке | `LookupError(f"Missing translation: '{full_key}' for lang '{lang}'")` |
| `_("key", wrong_kwarg=...)` — в шаблоне нет плейсхолдера | `str.format` игнорирует лишние аргументы |
| `_("key")` — в шаблоне есть `{name}`, но `name` не передан | `KeyError` от `str.format` |
| `translator.set_language(invalid)` | `ValueError` со списком доступных |
| Невалидный YAML при `set_language` | `ValueError(f"Invalid YAML in {file}: {e}")` |
| Коллизия лист/ветка при flatten | `ValueError` с указанием пути |
| `Translator(locales_dir)` — директория отсутствует | `FileNotFoundError` при init |
| `app.translator.available_languages()` пусто при `app.start()` | `RuntimeError("No language catalogs in ...")` |
| Библиотечный `_()` до `bind()` | `RuntimeError("Translator not bound")` |

Все ошибки конфигурации и загрузки — fail fast на старте приложения. Рантайм-ошибки — `LookupError`/`KeyError`, без скрытых fallback.

## Структура файлов

### В пакете `mypycli`

```
mypycli/
  i18n/
    __init__.py        # реэкспорт: Translator
    translator.py      # class Translator
    internal.py        # _active, bind, _ (только для библ. кода)
    loader.py          # yaml-парсинг + flatten + валидация коллизий
    detect.py          # _parse_lang_env
  locales/
    en.yml
    ru.yml
    zh.yml
  cli/
    commands/
      locales.py       # handlers init/sync/check
    standalone.py      # entry point mypycli для dev-команд
  application.py       # + translator param, + _resolve_and_apply_language
  console/
    builtin.py         # cmd_db_set: pre-validate language, post-set restart-hint
  database/
    schema.py          # + language: str = ""
```

### В проекте пользователя (после `mypycli locales init`)

```
myapp-project/
  locales/
    en.yml
    ru.yml
  myapp/
    main.py
    ...
```

### pyproject.toml

```toml
[project]
dependencies = [
    "pyyaml>=6",
    # ...
]

[project.scripts]
mypycli = "mypycli.cli.standalone:main"
```

## Тестирование

### Юнит

`tests/i18n/`:

- `test_translator.py`
  - `__init__` сканирует *.yml → `available_languages()` возвращает корректное множество.
  - `set_language(valid)` грузит, `__call__` отдаёт строки.
  - `set_language(invalid)` → `ValueError`.
  - `__call__` до `set_language` → `RuntimeError`.
  - `__call__` на отсутствующий ключ → `LookupError`.
  - `__call__` с kwargs: подстановка в `{name}`, лишние игнорируются, нехватка — `KeyError`.
- `test_loader.py`
  - Nested → flat dotted.
  - Коллизия лист/ветка → `ValueError`.
  - Невалидный yaml → `ValueError`.
- `test_detect.py`
  - `en_US.UTF-8` → `en`, `ru_RU` → `ru`, `C` → `""`, `""` → `""`.
- `test_internal.py`
  - `bind` + `_()` работает; автопрефикс `mypycli.` виден в вызове underlying Translator.
  - Без `bind` → `RuntimeError`.

### Интеграционные

`tests/integration/`:

- `test_application_i18n.py`
  - Первый старт, пустой `db.language`, `LANG=ru_RU` → на диске `db.language == "ru"`, activate ru.
  - Повторный старт — значение сохраняется.
  - `LANG=fr`, fr.yml нет → `db.language == "en"`, warning в лог.
  - `db.language == "ru"`, потом ru.yml удалён → перерезолв на старте, `db.language` перетирается.
- `test_cli_locales.py`
  - `locales init` в tmpdir → создаёт все файлы из ресурсов библиотеки.
  - Повторный `locales init` → skip существующих.
  - `locales sync` — мержит `mypycli:`, пользовательские ключи нетронуты.
  - `locales check` — exit 0 на согласованном, exit 1 на пропуске/out-of-sync с корректным текстом.
- `test_db_set_language.py`
  - `db set language ru` при наличии ru.yml → персист + restart-подсказка.
  - `db set language xx` → красная ошибка, БД не изменяется.
  - `db set debug true` → персист + restart-подсказка.

## Вне скоупа

- Плюрализация и CLDR-правила.
- Runtime-смена языка без перезапуска.
- Модули-пакеты с собственными переводами через ресурсы (автор модуля документирует используемые ключи; пользователь добавляет их в свои каталоги руками).
- AST-скан кода в `locales check` (заменён на сверку каталогов между собой).
- RTL/BiDi-поддержка, специальные правила форматирования чисел/дат по локали.
